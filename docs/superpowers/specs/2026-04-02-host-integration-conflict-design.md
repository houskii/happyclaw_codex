# 宿主来源冲突仲裁设计

## 背景

宿主来源统一接入后，HappyClaw 会同时读取：

- provider 绑定来源：`~/.claude`、`~/.codex`
- 自定义来源：例如 `~/.agent`

这解决了“能否接入”的问题，但没有解决“同名能力冲突时到底生效哪一份”的问题。

当前系统默认采用“来源顺序覆盖”策略：后面的来源覆盖前面的同名项。这个策略对同步器实现足够简单，但对用户不透明：

- 用户无法知道当前 Skill / MCP Server 实际来自哪个来源
- 用户无法在不改来源顺序的前提下固定某一份实现
- 页面无法解释为什么某个同名项在不同机器上的行为不同

本设计目标是把“来源接入”与“版本仲裁”显式拆开。

## 目标

- 保留现有“宿主来源清单”作为接入层
- 在 `Skills` / `MCP Servers` 控制页增加冲突识别与版本仲裁
- 对每个冲突项支持显式选择生效版本
- 允许用户恢复为“自动（按来源优先级）”
- 不改变现有最终消费链路，只改变同步决策层

## 非目标

- 不做全局统一的版本控制开关
- 不做工作区级冲突仲裁
- 不在 Provider 管理页里处理单个 Skill / MCP 的版本选择
- 不做复杂 diff 可视化（第一阶段只展示来源和当前生效版本）

## 设计原则

### 1. 来源接入和版本仲裁分层

- Provider 管理页负责：
  - 哪些来源接入
  - 哪些来源允许暴露 Skills / MCP
- Skills / MCP 页面负责：
  - 同名项是否冲突
  - 当前生效版本来自哪里
  - 是否需要手动固定版本

### 2. 自动优先，手动覆盖

默认仍然使用来源优先级自动决策，避免给无冲突场景增加额外负担。

只有当同名项来自多个来源时，页面才需要暴露“冲突”和“版本选择”。

### 3. 项级别仲裁，不做来源级仲裁

版本控制是针对单个 Skill / 单个 MCP Server 的，而不是针对整条来源的。

原因是：

- 用户可能只想固定某一个冲突 Skill
- 同一个来源下的其他条目仍应该继续按自动规则处理

## 方案对比

### 方案 A：继续使用来源顺序覆盖

优点：
- 实现最简单
- 不需要新增状态模型

缺点：
- 用户不可见、不可控
- 只能通过调整来源顺序间接控制结果
- 无法回答“当前到底用的是哪一份”

### 方案 B：来源级版本锁定

优点：
- UI 简单
- 用户可以声明“优先使用某个来源”

缺点：
- 粒度过粗
- 一个来源里多个条目会被整体绑死
- 不适合混合使用 `~/.claude`、`~/.codex`、`~/.agent`

### 方案 C：项级冲突仲裁（推荐）

优点：
- 精度足够高
- UI 心智清楚
- 不影响无冲突路径
- 容易和现有来源模型组合

缺点：
- 需要新增 override manifest
- 需要在 Skills / MCP 页面额外展示冲突信息

## 推荐方案

采用方案 C：项级冲突仲裁。

默认规则仍为“自动（按来源优先级）”，但对于同名项，允许在各自控制页显式选择某个来源版本作为强制生效版本。

## 数据模型

新增项目级冲突仲裁配置：

```ts
type HostIntegrationConflictOverride = {
  kind: 'skill' | 'mcp';
  itemId: string;
  mode: 'auto' | 'pinned';
  pinnedSourceId?: string;
};
```

项目级配置结构：

```ts
type HostIntegrationConflictSettings = {
  overrides: HostIntegrationConflictOverride[];
};
```

约束：

- `mode = 'auto'` 时，不需要 `pinnedSourceId`
- `mode = 'pinned'` 时，必须存在 `pinnedSourceId`
- `pinnedSourceId` 必须指向当前仍启用且暴露对应能力的来源

## 冲突识别

### Skills

当同一个 `skillId` 在多个来源中都被扫描到时，认为是冲突项。

页面展示：

- Skill 名称
- 候选来源列表
- 当前生效来源
- 当前模式：`自动` / `固定`

### MCP Servers

当同一个 `serverId` 在多个来源中都被扫描到时，认为是冲突项。

页面展示：

- Server ID
- 候选来源列表
- 当前生效来源
- 当前模式：`自动` / `固定`

## 决策规则

同步器最终选择实际生效版本时，按以下顺序：

1. 如果该项存在 `mode = pinned` 的 override：
- 优先使用 `pinnedSourceId`
- 若来源不存在、被禁用或未开放对应能力，则回退到 `auto`

2. `auto` 模式：
- 按来源优先级选择最后一个候选项
- 第一阶段继续沿用现有来源顺序逻辑

## 前端设计

### Skills 页面

新增“冲突项”视图：

- 正常项继续按当前方式展示
- 冲突项增加冲突标记
- 点开后显示：
  - 候选来源
  - 当前生效来源
  - 版本选择器：
    - `自动（按来源优先级）`
    - `固定到 <来源>`

### MCP Servers 页面

同样新增“冲突项”视图。

MCP 页面只仲裁 MCP 的冲突，不复用 Skills 的 override。

### Provider 管理页

不增加单项版本控制功能。

这里仍然只展示：
- 来源是否接入
- 来源是否暴露 Skills
- 来源是否暴露 MCP

## 后端设计

### 配置持久化

将冲突 override 作为新的系统设置字段保存。

建议字段名：
- `hostIntegrationConflictSettings`

### 扫描结果增强

共享扫描器需要返回“候选来源映射”，而不是只返回最终汇总结果。

至少要能得到：

```ts
type SkillConflictCandidate = {
  itemId: string;
  sourceId: string;
  sourceLabel: string;
  path: string;
};
```

MCP 同理。

### 新增接口能力

- 获取冲突概览
- 更新某个冲突项的 override
- 重新计算当前生效版本

第一阶段可以放在现有 Skills / MCP API 中附带返回，不强制拆独立路由。

## 错误处理

- 如果 `pinnedSourceId` 指向失效来源，不报致命错误
- 页面提示“固定来源当前不可用，已临时回退自动模式”
- 后端可选择在下一次保存时清理无效 override

## 测试策略

### 后端

- 多来源同名 Skill 冲突识别
- 多来源同名 MCP 冲突识别
- `auto` 模式仲裁测试
- `pinned` 模式仲裁测试
- pinned 来源失效后的自动回退测试

### 前端

- 冲突项展示测试
- 自动 / 固定切换测试
- 固定来源失效提示测试

## 分阶段实施

### 第一阶段

- 新增 override manifest
- 同步器支持 `auto / pinned` 决策
- Skills / MCP 页面展示冲突项和版本选择

### 第二阶段（可选）

- 冲突项 diff 展示
- 冲突筛选视图
- 批量恢复自动模式
